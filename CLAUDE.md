# Assistant — телеграм-бот для сбора задач

Личный бот-трекер задач в Telegram. Пишете боту текст — он сохраняется в базе данных.
Реализован **без собственного кода** — как сценарий (workflow) в n8n, который уже
развёрнут в проекте [[n8n-hub]] (см. `../n8n-hub/CLAUDE.md`).

Общие правила вайбкодинга — в `../CLAUDE.md` (корень VS_hub). Этот файл — только
специфика проекта Assistant.

## Статус

MVP + привязка задач к совещаниям. Список задач (просмотр в боте), отметка
выполнения, напоминания по срокам — следующие шаги, каждый отдельным заходом.

## Как это работает

1. Пользователь (только владелец, Telegram ID `325113969`) пишет сообщение боту
   `@AI_kuzkin_bot`.
2. Сценарий n8n на сервере Timeweb (5.129.228.135) ловит сообщение через
   Telegram-триггер, проверяет, что писал именно владелец (иначе — тишина).
3. Дальше сообщение разбирается на тип:
   - **`/meeting_start [название]`** — начинает совещание. Без названия — имя
     ставится автоматически по дате/времени («Совещание 22.07.2026 14:30»).
   - **`/ea`** — быстрый старт типового совещания «ЕА нед» (без набора текста).
   - **`/ai`** — быстрый старт типового совещания «АИ нед».
   - **`/work`** — быстрый старт типового совещания «Work».
   - **`/meeting_end`** — завершает текущее открытое совещание (проставляет
     `ended_at`). Если открытых совещаний нет — бот отвечает «Активных совещаний нет».
   - **любой другой текст** — обычная задача: сохраняется в таблицу `tasks`; если
     на момент отправки есть незавершённое совещание — задача автоматически
     привязывается к нему (`meeting_id`), и бот об этом сообщает в ответе.
   - **любая другая команда** (начинается на `/`, но не распознана) — игнорируется
     молча, чтобы не засорять список задач служебными командами.
4. Бот отвечает подтверждением («Задача добавлена ✅», «Совещание «…» начато/завершено»)
   и показывает под полем ввода кнопки-ярлыки: «/ea», «/ai», «/work», «/meeting_end» — чтобы
   не набирать команды вручную. Нажатие на кнопку отправляет её текст как обычное
   сообщение (это стандартное поведение Telegram reply-клавиатуры).

Никакого собственного сервера/кода у бота нет — вся логика в визуальном
конструкторе n8n (веб-интерфейс: http://5.129.228.135:5678, редактировать сценарий
можно там же). Правки в схему вносятся через n8n API или через веб-интерфейс n8n
(не через файлы этого репозитория).

### Схема совещаний в базе

- Таблица `meetings` (id, name, started_at, ended_at) — одна строка на совещание;
  `ended_at IS NULL` означает, что совещание ещё открыто (максимум одно открытое
  совещание одновременно — «Найти активное» берёт самое свежее с `ended_at IS NULL`).
- В `tasks` добавлена колонка `meeting_id` (nullable, внешний ключ на `meetings.id`) —
  задачи без активного совещания сохраняются с `meeting_id = NULL`.

## Схема сценария (n8n)

```
Telegram Trigger → Owner only (IF: from.id = 325113969) → Classify (Code: определяет тип сообщения)
  → Is meeting_start? → Insert meeting (Postgres) → Reply start
  → Is meeting_end?   → End meeting (Postgres)     → Reply end
  → Is task?          → Find active meeting (Postgres) → Save task (Postgres) → Reply task
```

- «Owner only» отсекает чужие сообщения — бот публичный (findable по @username),
  но реагирует только на владельца.
- «Classify» (JS-код) разбирает текст: `/ea`, `/ai`, `/meeting_start [имя]`,
  `/meeting_end` или обычная задача — распознаются по точным фразам, не по смыслу.
- Все запросы к базе — через параметризованные SQL-запросы Postgres-узла
  (`options.queryReplacement`, значения передаются массивом, а не склейкой строк) —
  исключает SQL-инъекции через текст задачи или название совещания.
- Узлы «End meeting» и «Find active meeting» помечены `alwaysOutputData: true` —
  если открытого совещания нет, они всё равно отдают пустой результат дальше по
  цепочке, а не обрывают выполнение.

## Известные нюансы

- Сервер общий с [[n8n-hub]] — там же живёт VPN-контейнер `amnezia-awg`. Любые
  Docker-операции (пересоздание контейнеров, новые сети) могут на секунды сбить его
  iptables. После любых изменений на этом сервере — `docker restart amnezia-awg` и
  проверка NAT-правила (см. `../n8n-hub/CLAUDE.md`).
- Вебхук Telegram работает через `https://hub.каскад.team` (Cloudflare перед сервером
  терминирует HTTPS) — то же, что использует бот Acifrui.
- Смотреть данные в базе можно так:
  ```
  ssh root@5.129.228.135 "docker exec tasks-db psql -U tasksbot -d tasksdb -c 'SELECT * FROM tasks ORDER BY id DESC LIMIT 20;'"
  ```

## Админка (просмотр, статусы, канбан)

Отдельный публичный сайт на GitHub Pages, две вкладки:
- **Список** — задачи, сгруппированные по совещанию (с датой), с кнопкой удаления.
- **Канбан** — три колонки (Открыта / В работе / Готово), карточки перетаскиваются
  мышью между колонками (меняет статус), у каждой карточки есть кнопка удаления.

|---|---|

| Backend (API) | n8n workflow id `bGf3PucODMr3T9Xi` («Assistant — API для админки задач») |
| Токен доступа | см. `../ДОСТУПЫ.md`, раздел «Телеграм-бот задач (Assistant)» — без него любой запрос отвечает 403 |

### API-эндпоинты (все — GET с `?t=токен`, без preflight/CORS-сложностей)

| Путь | Параметры | Действие |
|---|---|---|
| `/webhook/assistant-tasks` | — | Список задач (JSON), `LEFT JOIN meetings` |
| `/webhook/assistant-tasks-status` | `id`, `status` (`open`/`in_progress`/`done`) | Меняет статус; `done` дополнительно проставляет `done_at = now()`, любой другой статус сбрасывает `done_at` в NULL |
| `/webhook/assistant-tasks-delete` | `id` | Удаляет задачу безвозвратно (без корзины) |
| `/webhook/assistant-tasks-mark` | `id`, `context`, `auto_run` (`true`/`false`) | Ставит/снимает пометку 🤖 для ночного агента, сохраняет контекст, сбрасывает `run_status`/`run_result` |
| `/webhook/assistant-tasks-queue` | — | Для облачного агента: задачи с `auto_run=true` и ещё не обработанные |
| `/webhook/assistant-tasks-runresult` | `id`, `run_status`, `run_result` | Для облачного агента: записать результат прогона задачи |

Базовый адрес: `https://hub.каскад.team`. Проверка токена — в каждом Webhook-узле
отдельным IF-узлом (сравнение строк), не в коде сайта — подделать запрос без
токена нельзя, даже зная структуру API.

### Как это работает
- Страница `index.html` берёт токен из своей же ссылки (`?t=...`) один раз и
  использует его для всех запросов к API.
- Вкладки переключаются на клиенте (без перезагрузки) — один и тот же список задач
  рисуется по-разному для «Список» и «Канбан».
- Удаление и смена статуса — сразу оптимистично меняют вид на странице, запрос к
  API уходит фоном (без ожидания ответа перед перерисовкой).
- Правки в API — через n8n (веб-интерфейс или API), правки в вид страницы —
  через `index.html` в этом репозитории (`git push` в main публикует сам, GitHub
  Pages, ветка main, корень).

### Известные ограничения
- Удаление — без подтверждения на сервере и без возможности восстановить (только
  браузерный `confirm()` перед отправкой запроса).
- Перетаскивание карточек — только мышью (HTML5 drag-and-drop), на сенсорных
  экранах (телефон/планшет) не работает — там пока можно менять статус только
  через список задач в самом Telegram-боте (когда появится такая команда) или
  правкой в базе.

## Ночной облачный агент (авто-выполнение помеченных задач)

Каждую задачу можно пометить кнопкой 🤖 в админке (просит ввести подробный
контекст: какой проект, что сделать). Раз в сутки, в 01:00 по Москве, отдельный
облачный агент (routine на claude.ai, id `trig_01VYsU3DimEKWD8ZUPFNiMTp`, cron
`0 22 * * *` UTC) забирает такие задачи и пытается их выполнить.

**Важно: это ДРУГОЙ агент**, не связанный с обычными сессиями Claude Code — у
него нет доступа к этому компьютеру, серверу Timeweb (SSH/база данных напрямую)
и приватным репозиториям. Только к перечисленным ниже публичным репозиториям и к
API этого проекта через интернет.

### Что он делает
1. GET `.../webhook/assistant-tasks-queue?t=токен` — получает задачи с `auto_run=true`
   и `run_status` в (`NULL`, `queued`).
2. Для каждой — определяет проект по тексту/контексту, читает CLAUDE.md проекта,
   вносит правку **в отдельной git-ветке** (`auto-task-<id>-...`), пушит ветку,
   открывает **Pull Request** — и всё, дальше стоп: **никогда сам не сливает PR,
   не деплоит и не публикует** (ни `clasp deploy`, ни `netlify deploy`, ни merge
   в main). Слияние и деплой — вручную, пользователем, после ревью (можно попросить
   обычную сессию Claude Code «глянь PR по задаче N» — она прочитает диф и даст
   короткую сводку понятным языком, решать пользователю).
3. GET `.../webhook/assistant-tasks-runresult?t=токен&id=X&run_status=...&run_result=...`
   — пишет результат обратно (обязательно для каждой обработанной задачи):
   `pr_opened` (+ ссылка на PR), `needs_clarification` (+ что неясно) или `failed`
   (+ причина). Статус и ссылка видны в админке рядом с задачей.

### Ограничение: приватные репозитории недоступны
Агент подключён только к публичным репозиториям: `shahmatka`, `shahmatka-sb5`,
`parking-admin`, `cc-shahmatka`, `supply-form`, `assistant-tasks`. Приватные —
`contracts-control` (Контроль договоров), `daily-digest`, `acifrui` — облачная
среда claude.ai не пускает (пробовал подключить, отказ прав доступа). Задачи по
этим проектам агент помечает `needs_clarification` с причиной «нет доступа к
репозиторию» — их можно взять только в обычной сессии Claude Code.

### Как пометить задачу
В админке — кнопка 🤖 у задачи → вводите в подсказке контекст (обязательно
указать, к какому проекту относится задача — иначе агент не рискнёт угадывать
и тоже отметит needs_clarification). Повторное нажатие снимает пометку.

## Планы (не сделано)

- `/list` — показать открытые задачи (в том числе сгруппированные по совещанию/дате).
- Отметка задачи выполненной (кнопка или команда).
- Напоминания по дедлайну.
- Сообщение при `/meeting_start`, если уже есть открытое совещание (сейчас можно
  открыть второе поверх первого — новые задачи уйдут во второе, первое останется
  «висеть» открытым до отдельного `/meeting_end`).
